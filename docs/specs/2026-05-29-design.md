# babitalk-front-hermes — Design Spec

- **Date**: 2026-05-29
- **Author**: 권단비 (FE Admin)

## 1. Purpose & Motivation

babitalk 프론트엔드 모노레포(`babitalk_monorepo_frontend`)에 특화된, **로컬에서 동작하는 AI Q&A 툴**을 만든다.

사용자(주로 FE 개발자)가 다음과 같이 질의하면 코드 근거를 들어 답한다:

- 스크린샷 + URL을 주고 "이 페이지에서 어떤 API를 호출하나요?"
- 특정 함수/액션을 주고 "이거 바꾸면 어디 영향이 가나요?"
- 자유 형태 Q&A: 로직 설명, 의심 부분 추적

목표: **Hermes 같은 사내 AI 어시스턴트의 로컬 MVP**. 향후 슬랙봇으로 확장한다 (별도 프로젝트).

## 2. Scope

### In Scope (MVP)
- 분석 대상: `babitalk_monorepo_frontend` 전체 FE (Admin + Client + shared packages)
- 입력: (선택) 스크린샷 이미지, (선택) URL, (필수) 자연어 질문
- 출력: 마크다운 답변 + 파일:라인 근거
- UX: CLI (단발 `hermes ask` + 대화형 `hermes chat`)
- 코드 정적 분석 + LLM 추론 결합

### Out of Scope
- 백엔드 코드 분석 (FE 외부 추론 불가)
- 슬랙봇/웹 UI/VSCode extension (다음 프로젝트)
- 멀티 사용자 / 클라우드 호스팅
- 런타임 동작 추적, 동적 라우팅 path 추적
- 임베딩 기반 RAG (정적 그래프로 충분, 부족 시 Phase 2)

## 3. Key Decisions

| 영역 | 결정 | 근거 |
|------|------|------|
| 분석 범위 | Monorepo 전체 FE | 사용자 동선이 도메인 경계를 넘나듦. Admin 한정은 협소 |
| 스크린샷 매핑 | URL/라우트 메타 + Vision 결합 | 라우터 정적 파싱으로 후보 좁히고 vision으로 영역 식별. 정확도/구현 난이도 균형 |
| 사이드 이펙트 분석 | 정적 그래프 (ts-morph) + LLM 설명 | 결정론적 영향도 그래프로 정확도 확보, LLM은 해설 담당 |
| MVP 질문 유형 | 자유 Q&A | API 매핑 + 영향도 + 일반 코드 질문까지 |
| 아키텍처 | 코어 라이브러리 + CLI 어댑터 | 슬랙봇 확장 시 코어 재사용 |
| 인덱스 저장소 | JSON (`.hermes/index.json`) | 단순함 우선. 성능 문제 발생 시 SQLite 전환 |

## 4. Architecture

### 4.1 패키지 구조

```
babitalk-front-hermes/
├── packages/
│   ├── core/                    # @hermes/core (재사용 라이브러리)
│   │   ├── src/
│   │   │   ├── indexer/         # 코드베이스 → JSON 인덱스
│   │   │   ├── retriever/       # JSON 인덱스 → 쿼리 API
│   │   │   ├── agent/           # Claude tool-use loop
│   │   │   ├── tools/           # agent에 노출되는 tools
│   │   │   └── types/
│   │   └── package.json
│   └── cli/                     # @hermes/cli (CLI 어댑터)
│       ├── src/
│       │   ├── commands/        # index, ask, chat, init
│       │   └── index.ts
│       └── package.json
├── tests/
│   └── fixtures/sample-repo/    # 미니 모노레포 fixture
├── docs/specs/
└── pnpm-workspace.yaml
```

`.hermes/` (런타임 캐시)는 **분석 대상 모노레포의 root**에 생성한다 (대상별 격리). gitignore 대상.

### 4.2 데이터 흐름

```
[User]
   │ hermes ask --image foo.png --url <URL> -q "<질문>"
   ▼
[CLI command (ask)]
   │  - 인덱스 fresh 확인
   │  - 입력 전처리 (이미지 base64 등)
   ▼
[Agent (core)]
   │  System prompt + multi-modal user message
   │  ┌─ tool_use ──> Retriever 메서드 호출
   │  └─ tool_result ──> 다음 iteration
   ▼
[stdout: Markdown 응답]
```

### 4.3 스택

| 영역 | 선택 |
|------|------|
| 언어 / 런타임 | TypeScript / Node.js 20+ |
| LLM | Claude Sonnet 4.6 (vision + tool use) |
| SDK | `@anthropic-ai/sdk` (직접 agent loop 구성) |
| 정적 분석 | `ts-morph` |
| CLI | `commander` + `@inquirer/prompts` |
| 패키지 매니저 | pnpm workspaces |
| 테스팅 | Vitest |

## 5. Indexer

ts-morph로 코드베이스를 스캔해 다음 5종을 JSON으로 추출:

### 5.1 추출 항목

1. **routes** — 라우터 설정 파일 파싱 → `{ app, path, componentPath }[]`
2. **components** — 각 React 컴포넌트의 import + JSX 사용 → `{ imports[], renders[] }`
3. **apiCalls** — RTK Query `createApi`의 endpoints + saga/axios/fetch 호출 → `{ id, method, url, definedIn, kind, usedBy[] }`. `kind`는 `"rtk-query" | "saga" | "axios" | "fetch"` enum
4. **reduxUsage** — slice actions/selectors 정의 및 사용처
5. **fileGraph** — import 그래프 + **역방향(importedBy) 인덱스**

### 5.2 JSON 스키마 (요약)

```json
{
  "version": "1",
  "indexedAt": "<ISO 8601>",
  "monorepoRoot": "<absolute path>",
  "apps": ["admin", "client"],
  "routes": [...],
  "components": {...},
  "apiCalls": [...],
  "reduxUsage": { "actions": [...], "selectors": [...] },
  "fileGraph": {...}
}
```

### 5.3 명령

```bash
hermes index                 # 전체 재인덱싱
hermes index --app admin     # 특정 앱만
```

### 5.4 비고

- 대상 사이즈에 따라 5~20MB 예상. 메모리 로드 OK 수준
- TS 파싱 실패 파일은 skip + 로그. 인덱싱은 계속 진행
- 인크리멘털 인덱싱은 Phase 2

## 6. Retriever + Agent Loop

### 6.1 Retriever (내부)

`@hermes/core/retriever`가 JSON 인덱스를 메모리에 로드하고 쿼리 메서드 노출:

```ts
class Retriever {
  routeFromUrl(url: string): Route | null;
  componentsForRoute(route: Route): ComponentNode[];
  apiCallsInComponent(componentPath: string): ApiCall[];
  apiCallsInSubtree(rootComponent: string): ApiCall[];
  dependentsOf(filePath: string): string[];
  reduxUsageOf(actionOrSelector: string): UsageInfo;
  searchByKeyword(keyword: string): SearchHit[];
  readFileSlice(path: string, range?: LineRange): string;
}
```

### 6.2 Agent에 노출되는 Tools (6개)

| Tool | 입력 | 출력 |
|------|------|------|
| `route_from_url` | `url` | 매칭된 라우트 + 컴포넌트 path |
| `list_apis_in_component` | `componentPath`, `recursive?` | API 호출 목록 |
| `find_dependents` | `filePath \| symbolName` | 역참조 목록 |
| `find_redux_usage` | `actionOrSelector` | 정의 + 사용처 |
| `read_file` | `filePath`, `lineRange?` | 파일 내용 |
| `search_code` | `keyword` | 인덱스 기반 키워드 검색 |

→ Tool 개수를 의도적으로 6개로 제한 (LLM의 tool 선택 정확도 보호).

### 6.3 Vision 전처리

별도 전처리 없음. 이미지를 multi-modal 메시지로 그대로 첨부하고 Claude가 영역 식별 + tool 가설 검증 수행.

### 6.4 응답 형식 (system prompt에서 강제)

```markdown
## 답변
<자연어 답변>

## 근거
- <파일경로>:<라인> — <설명>

## 주의/사이드 이펙트 (해당 시)
- ...
```

### 6.5 CLI 전체 명령

```bash
hermes init                                             # 첫 셋업: ~/.hermes/config.json + .env 가이드
hermes index [--app admin|client] [--force-reindex]     # 인덱스 빌드/갱신
hermes ask --image <path> --url <URL> -q "<질문>"        # 단발 질의
hermes chat [--image <path>] [--url <URL>]              # 대화형 REPL

# 공통 옵션
--verbose                                               # tool call 흐름 표시
--debug                                                 # LLM raw 요청/응답 dump
--yes-i-know                                            # 50K input token 초과 시 진행
```

## 7. Error Handling & Limits

### 7.1 실패 시나리오

| 시나리오 | 대응 |
|---------|------|
| 인덱스 없음 | "`hermes index`부터 실행" 안내. 자동 실행 X |
| 인덱스 stale (>7일) | 경고 + 진행. `--force-reindex` 제공 |
| URL 매칭 실패 | tool이 null 반환 → agent가 스크린샷만으로 추론 fallback |
| 스크린샷 없음 | 허용. 텍스트 질문만으로 동작 |
| LLM rate limit | 지수 backoff 3회. 실패 시 명시 |
| 파일 100KB+ | 자동 truncate + 경고 |
| `ANTHROPIC_API_KEY` 누락 | `hermes init` 가이드 안내 |
| ts-morph 파싱 실패 | 해당 파일 skip + 로그 |

### 7.2 명시적 한계 (README에 기재)

- 백엔드 영향도 추론 불가
- 정적 분석만 — 런타임 분기 미반영
- 동적 import / 문자열 path 라우팅 누락 가능
- `node_modules` 미인덱싱
- 100% 정확도 비보장. 답변은 반드시 파일:라인 근거로 검증

### 7.3 Observability

```bash
hermes ask ... --verbose      # tool call 흐름 표시
hermes ask ... --debug        # LLM raw 요청/응답 dump
```

### 7.4 비용 가드

- 매 요청 직전 token estimate 출력
- 50K input token 초과 시 `--yes-i-know` 플래그 요구
- 일일 누적 비용 트래킹 (`~/.hermes/usage.log`)

## 8. Testing

| 레이어 | 도구 | 대상 |
|--------|------|------|
| Unit | Vitest | Retriever 쿼리, 인덱서 파서 |
| Integration | Vitest + fixture | `tests/fixtures/sample-repo/` 인덱싱 → tools 호출 → 결과 검증 |
| Agent | recorded fixtures | Anthropic SDK mock으로 tool loop 회귀 테스트. 실제 API 호출 X |
| E2E (수동) | 실제 babitalk 모노레포 | 골든 질문 5~10개 세트로 정확도 회귀 모니터링 |

### 골든 질문 예시

1. 특정 URL 페이지에서 호출하는 API 전체 리스트
2. 특정 Redux action을 dispatch하는 파일 전체
3. 특정 API 함수 시그니처 변경 시 영향받는 컴포넌트 전체

## 9. Distribution

- 별도 git 저장소: `babitalk-front-hermes`
- 분석 대상 `babitalk_monorepo_frontend`와 **분리** (관심사 분리)
- MVP는 npm publish 안 함 — 로컬 빌드 + `pnpm link --global`

### 환경 설정

```bash
hermes init   # ~/.hermes/config.json + .env 가이드 생성
```

`.env` 항목:
- `ANTHROPIC_API_KEY`
- `HERMES_MODEL` (기본 `claude-sonnet-4-6`)
- `HERMES_TARGET_ROOT` (분석 대상 모노레포 absolute path)

## 10. Milestones

| 마일스톤 | 산출물 | 예상 |
|---------|-------|------|
| M1: 인덱서 기본 | routes + components + apiCalls(RTK Query) 추출 + JSON 출력 | 3~4d |
| M2: 인덱서 확장 | reduxUsage + fileGraph(역방향) + saga/axios API 호출 | 2~3d |
| M3: Retriever + Tools | 6개 tool 구현 + 단위 테스트 | 2d |
| M4: Agent loop | SDK + tool loop + system prompt | 2d |
| M5: CLI | `init`, `index`, `ask`, `chat` + verbose/debug | 2d |
| M6: 골든 질문 + 정확도 측정 | GT 페어 + 첫 측정 | 2d |

**합계 ~13~15 영업일 (약 3주)**. M1~M3 종료 시점부터 API 매핑 dogfooding 가능.
